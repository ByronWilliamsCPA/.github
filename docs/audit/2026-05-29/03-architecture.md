# 03 - Architecture and Structure

The repo runs two parallel Python workflow sets: 22 reusable `workflow_call` workflows in `.github/workflows/python-*.yml` and 15 template gallery files in `workflow-templates/`. The intended layering (reusable = implementation, template = thin caller) holds for 9 of 15 templates but is violated by 5 that re-implement jobs inline, and one template (`python-slsa`) calls a reusable that does not expose `workflow_call` at all, so it is a broken caller. Action SHA pinning is consistent across all 34 files (no SHA drift), but Python-version lists and ADR locations are fragmented.

**ARCH-01: python-slsa template calls a reusable that has no workflow_call trigger (broken at @v1)**
Severity: Critical; Effort: S (add `workflow_call:` block, or repoint template). Basis: single-file trigger change plus input mapping.
Evidence: `workflow-templates/python-slsa.yml:73` does `uses: ByronWilliamsCPA/.github/.github/workflows/python-slsa.yml@6f71aecae2...  # v1`, but `.github/workflows/python-slsa.yml:66-67` declares only `on: workflow_dispatch:` (no `workflow_call`). The reusable is even named "SLSA Provenance (Template)" (`.github/workflows/python-slsa.yml:62`). Any repo adopting this template fails at workflow load with "workflow does not have workflow_call trigger."
Recommendation: Add a `workflow_call:` trigger (with inputs) to `.github/workflows/python-slsa.yml` and rename it off "(Template)", or convert the template to inline steps.

**ARCH-02: Template/reusable layering is inconsistent: 5 templates re-implement steps instead of calling reusables**
Severity: High; Effort: M (rewrite 5 templates as thin callers, verify input parity). Basis: per-file conversion plus drift reconciliation.
Evidence: 9 templates are correct thin callers (`python-pr-validation, python-reuse, python-container-security, python-sbom, python-slsa(broken), python-compatibility, python-fips-compatibility, python-mutation, python-scorecard`; e.g. `workflow-templates/python-scorecard.yml:32` is a clean `uses:` caller with one job). But these re-implement jobs inline despite a matching reusable existing: `python-ci.yml` (5 `runs-on`), `python-security-analysis.yml` (8 `runs-on`), `python-docs.yml` (5), `python-publish-pypi.yml` (3), `python-release.yml` (2), `python-codecov.yml` (1), `python-sonarcloud.yml` (1). Counts from `grep -c runs-on workflow-templates/*.yml`. Each inline template is a second copy of logic that already lives in `.github/workflows/python-<same>.yml`, doubling maintenance.
Recommendation: Convert the 5+ inline templates to the thin-caller pattern already used by `python-scorecard.yml`; reserve inline only where the gallery intentionally shows runnable steps (document that choice).

**ARCH-03: Template python-ci drifts from the reusable on Python versions and testing strategy**
Severity: High; Effort: M (reconcile version policy across files). Basis: requires deciding canonical version set, then editing.
Evidence: Template hardcodes `python-version: ["3.10", "3.11", "3.12", "3.13", "3.14"]` (`workflow-templates/python-ci.yml:138`). The reusable uses a tiered strategy with different defaults: `python-versions-pr` = `["3.11", "3.12"]` and `python-versions-comprehensive` = `["3.10", "3.11", "3.12", "3.13"]` (`.github/workflows/python-ci.yml:46,51`), and neither includes 3.14. The template also lacks the reusable's Vulture/LLM-governance/security jobs (`.github/workflows/python-ci.yml:16-19`). A user picking the gallery CI gets a different, simpler pipeline than the reusable it nominally mirrors.
Recommendation: Either repoint the template at the reusable (ARCH-02) so versions come from one source, or pin both to one documented version matrix.

**ARCH-04: Python-version lists are scattered as literals across files (no central source)**
Severity: Medium; Effort: M. Basis: introduce a single input/var and thread it through callers.
Evidence: Three distinct version-list literals appear independently: `["3.10".."3.14"]` in 1 file, `["3.10".."3.13"]` in 3 files, `["3.11", "3.12"]` in 1 file (counts via `grep -rl` over both dirs). Bumping the supported Python range requires editing multiple files with no shared definition. Contrast with action SHAs, which are uniform.
Recommendation: Centralize the supported-version matrix (a single reusable input default or a documented constants block) and have templates inherit it.

**ARCH-05: Action SHA pins are uniform (no drift)**
Severity: Low; Effort: S. Evidence: Each pinned action resolves to exactly one SHA across all 34 yml files: `harden-runner` 88 uses / 1 SHA, `checkout` 60/1, `setup-python` 26/1, `setup-uv` 17/1 (`grep -rho uniq -c`). `scripts/update-pinned-actions.sh` plus `scripts/fleet-audit-sha-pins.sh` evidently keep this consistent. No constant-drift finding here; this sub-area is clean.

**ARCH-06: Naming collision risk between repo-local CI and reusable exports (same dir, near-identical names)**
Severity: Medium; Effort: S (rename repo-local files or document the convention). Basis: file renames plus any branch-protection check-name updates.
Evidence: `.github/workflows/` mixes the repo's own CI with its reusable exports, distinguished only by a `python-` prefix and a "(Reusable)" name suffix. Pairs: `reuse.yml` (REUSE Compliance, no workflow_call) vs `python-reuse.yml` (REUSE Compliance (Reusable), workflow_call); `scorecard.yml` vs `python-scorecard.yml`; `security-analysis.yml` vs `python-security-analysis.yml`; `sonarcloud.yml` vs `python-sonarcloud.yml`; `pr-validation.yml` vs `python-pr-validation.yml`. Map: the unprefixed files are this repo's own self-CI (no `workflow_call`); the `python-*` files are the exported library (all `workflow_call=1`) per the per-file `name:`/trigger scan. Readers and tooling can easily edit the wrong one.
Recommendation: Document the "`python-` prefix = exported reusable, unprefixed = repo-local self-CI" rule in CLAUDE.md/AGENTS.md, or move repo-local CI to a clearly distinct naming set.

**ARCH-07: Two different ADR-001 documents in two directories; ADR coverage near-zero for major decisions**
Severity: Medium; Effort: S (consolidate ADR location, backfill 2-3 ADRs). Basis: doc moves plus short ADR writing.
Evidence: `docs/architecture/adr-001-scorecard-publish-results.md` ("Scorecard publish-results Deprecated") and `docs/planning/adr/adr-001-workflow-security-remediation-delivery.md` ("Workflow Security Remediation Delivery Strategy") share the ADR-001 id in two trees. The index (`docs/architecture/adr-000-index.md:7`) lists only the first and is unaware of the planning ADR. The two largest architectural decisions, the template/reusable split (this entire audit's subject) and the SHA-pinning + fleet-audit strategy, have no ADR. CLAUDE.md cross-references only `docs/architecture/adr-000-index.md`.
Recommendation: Pick one ADR directory, renumber to remove the duplicate ADR-001, and add ADRs for the template/reusable split and the SHA-pinning strategy.

**ARCH-08: CLAUDE.md scope statement understates script/test surface**
Severity: Low; Effort: S. Evidence: `.claude/CLAUDE.md` states the Bats suite "covers both `scripts/update-pinned-actions.sh` and `scripts/fleet-audit-sha-pins.sh`," but `scripts/` holds 7 shell scripts (`calculate-image-storage.sh, check-no-em-dash.sh, fleet-audit-sha-pins.sh, regenerate-checksums.sh, sync-secrets.sh, transfer-repos.sh, update-pinned-actions.sh`). The doc names 2 as the test targets; the other 5 are unmentioned, so their test status is undocumented.
Recommendation: Update CLAUDE.md to list all scripts and state which are under Bats coverage.
